
# Product Requirements Prompt (PRP) — [Status Marker] ✅

## 1. Project Context & Objectives

The iOS Safari WebKit engine requires absolute synchronicity between a user tap and media playback initialization. The current HTML5 `<audio>` keep-alive mechanism fails because fetching `silence.wav` over the network introduces a microscopic delay, causing iOS to reject the `.play()` promise and subsequently kill the Web Audio context when the screen locks. This PRP replaces the physical `.wav` file with an immediate, in-memory Base64 encoded string.

## 2. Technical Decisions & Dependencies

* **Base64 Audio Data URI:** We will use a heavily compressed, valid 44-byte RIFF WAV file containing a single frame of silence, encoded directly as a Base64 string.
* **Network Elimination:** `GenerativeAudioController.js` will no longer rely on the Vite `BASE` path or the network to establish the background audio loop.

## 3. Implementation Phases

### Phase 1: In-Memory Base64 Injection — [Status Marker] ✅

* **Objective:** Replace the physical asset fetch with a hardcoded Base64 data string inside the audio controller.
* **Instructions:**
1. **Open `src/engine/GenerativeAudioController.js**`.
2. At the top of the file, beneath the imports, declare the Base64 constant. This string is a perfectly valid, microscopic silent WAV file:
```javascript
const BASE64_SILENCE = "data:audio/wav;base64,UklGRigAAABXQVZFZm10IBIAAAABAAEARKwAAIhYAQACABAAAABkYXRhAgAAAAEA";

```


3. Locate the `startIOSKeepAlive()` function.
4. Update the `audio.src` assignment to use the Base64 string instead of the network path:
```javascript
// REMOVE THIS:
// audio.src = `${BASE}assets/silence.wav`;

// ADD THIS:
audio.src = BASE64_SILENCE;

```
