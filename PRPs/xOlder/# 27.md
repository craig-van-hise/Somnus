# Product Requirements Prompt (PRP) — [Status Marker] ✅

## 1. Project Context & Objectives

The current iOS Safari deployment suffers from a severely degraded UX when initiating the audio engine. The "Start Engine" button freezes without immediate visual feedback, leading to user confusion (Problem A). Furthermore, the underlying `GenerativeAudioController.bootEngine()` takes too long to execute because it duplicates network requests by attempting to fetch large audio assets during initialization, rather than utilizing the data already fetched during the app mount phase (Problem B).

This PRP addresses both issues by:

1. Injecting an immediate `'starting'` visual state into the React UI to provide snappy, scale-animated feedback.
2. Refactoring `assetLoader.js` to cache the downloaded `ArrayBuffer`s.
3. Refactoring `GenerativeAudioController.js` to bypass network calls entirely on boot, utilizing fast, native `decodeAudioData` on the cached array buffers.

## 2. Technical Decisions & Dependencies

* **UI State Transition:** We will leverage the existing `sessionStatus` string in `AppContext.jsx` by introducing a `'starting'` state. This ensures state exclusivity.
* **Asset Caching:** `assetLoader.js` will utilize a module-level variable (`globalAssetCache`) to hold the raw `ArrayBuffer`s fetched during the initial application load.
* **Memory-Safe Decoding algorithm:** In `bootEngine()`, we will extract the raw native `AudioContext`, slice a copy of the cached `ArrayBuffer` (to prevent Detached ArrayBuffer errors if decoded multiple times), and use `await nativeCtx.decodeAudioData()`. This resulting `AudioBuffer` will be wrapped in a `Tone.ToneAudioBuffer` and passed directly into `Tone.Player`.

### Phase 3: Zero-Latency Engine Boot & Decoding — [Status Marker] ✅

* **Objective:** Refactor `GenerativeAudioController.js` to eliminate network requests from the `bootEngine` sequence, significantly reducing the "Start Engine" initialization time.
* **Instructions:**
1. **Import Cache**: Import `globalAssetCache` from `../services/assetLoader.js` into `GenerativeAudioController.js`.
2. **Native AudioContext Extraction**: Inside `bootEngine()`, utilize the existing logic that extracts `nativeContext`. We need this to invoke `decodeAudioData`.
3. **Refactor Channel 4 (Nature Ambience) Initialization**: Remove the `fetch()` and `this.safeLoadBuffer()` calls entirely. Replace them with explicit memory decoding logic:
```javascript
// Phase 3 Zero-Latency Decoding Logic for Rain
try {
  if (globalAssetCache.rainBuffer && nativeContext) {
    // Slice the buffer to avoid Detached ArrayBuffer errors
    const rainData = await nativeContext.decodeAudioData(globalAssetCache.rainBuffer.slice(0));
    const toneRainBuffer = new Tone.ToneAudioBuffer(rainData);
    this.rainPlayer = new Tone.Player({
      url: toneRainBuffer,
      loop: true,
      fade: 2.0,
    });
    this.rainPlayer.loopStart = 2.0;
    this.rainPlayer.loopEnd = toneRainBuffer.duration - 2.0;
  } else {
     // Fallback to empty player if cache failed
     this.rainPlayer = new Tone.Player({ loop: true, fade: 2.0 });
  }
} catch (e) {
  console.warn("Rain decode error", e);
}

```


4. **Repeat for Ocean Waves**: Apply the exact same logic block for `oceanBuffer` to instantiate `this.wavesPlayer`.
5. **Remove `Promise.race([Tone.loaded()...])**`: Because we are explicitly providing fully decoded `AudioBuffer` objects synchronously to the `Tone.Player` constructors, we are no longer waiting on Tone.js's internal network loader. You can safely remove or shorten the `Tone.loaded()` await block at the end of `bootEngine()`.
